Heart Rate Sensor & F1 Safety Gear Study

Arduino Embedded C++ I²C Sensors MATLAB

A working pulse monitor built from scratch — an Arduino reading a MAX30102 pulse-oximeter over I²C, computing live BPM, and showing it on an OLED with a buzzer beat. I then used it as the springboard for a study on how Formula 1 safety gear affects a driver's heart rate.

At a glance

This was a hands-on circuits and embedded-programming project: wire a multi-component sensor system, get it talking over I²C, and write firmware that turns a raw optical signal into a stable, readable heart rate. With the device working, I built an applied study around it — exploring how stripping away different pieces of F1 safety gear might affect a driver's heart rate.

The build

The system centers on an Arduino Uno. A MAX30102 pulse-oximeter shines light into a fingertip and reads the reflected signal; the Arduino processes that signal to detect beats and compute BPM, then drives an SSD1306 OLED to display the value live. A buzzer pulses on each detected beat. The sensor and display share a single I²C bus (the A4/A5 lines), which keeps the wiring compact but means both devices must respond at different addresses on the same two wires.

Firmware

I started from provided starter code and optimized it for a stable reading — the raw sensor signal is noisy, so the real work is in beat detection and smoothing rather than just printing a number. The sketch initializes the sensor and display over I²C, samples the IR signal, detects peaks, converts the beat-to-beat interval into BPM, and refreshes the OLED.

// core loop: sample -> detect beat -> compute BPM -> display long irValue = particleSensor.getIR(); if (checkForBeat(irValue)) { long delta = millis() - lastBeat; // time since last beat lastBeat = millis(); beatsPerMinute = 60 / (delta / 1000.0); // rolling average smooths noisy single readings if (beatsPerMinute > 20 && beatsPerMinute < 255) { rates[rateSpot++] = (byte)beatsPerMinute; rateSpot %= RATE_SIZE; beatAvg = average(rates); tone(3, 1000, 50); // beep on each beat (buzzer on pin D3) } } display.print(beatAvg); display.print(" BPM");

Applied study: F1 safety gear & heart rate

With a working monitor in hand, I framed an engineering question around it: which piece of a Formula 1 driver's safety gear most affects their heart rate, and how does heart rate correlate with speed around a lap? The idea was that a driver's physical and psychological load — and therefore cardiovascular strain — depends on what protection they're wearing. I designed a four-condition protocol (full gear as control, then helmet only, racing suit only, and rib protectors only) measured at checkpoints around the Bahrain International Circuit, and analyzed the resulting BPM curves in MATLAB.

The modeled results pointed to an interesting interpretation: removing the racing suit alone produced the highest heart rate, while rib protectors alone kept it closest to the full-gear baseline. A plausible reading is psychological — rib protectors guard against several immediate impact modes (with structures, the steering wheel, the steering column), so their presence may ease a driver's stress more than the lightweight, fire-focused suit does.

Note on data: this study used simulated/modeled heart-rate data, not measurements from a real driver. The hardware build above is real and functional; the F1 analysis is a conceptual exploration of the kind of question the device could investigate with real subjects.

What I took away

The core win was getting real hardware to work end-to-end: debugging an I²C bus shared between two devices, taming a noisy biosignal into a stable reading, and writing firmware that does something useful in real time. It also pushed me to think past the build — to frame an actual research question the sensor could answer, which is exactly the bridge between a circuit and an experiment. A natural next step is to run the F1-gear protocol with real subjects and compare measured data against the model.
